Contents of an exception table.

The โno exceptionโ target for is stored as the last element of the underlying vector. It can be accessed through the normal_return and normal_return_mut functions. Exceptional catch clauses may be iterated using the catches and catches_mut functions. All targets may be iterated over using the all_targets and all_targets_mut functions.

Create new exception-table data.

This data represents the destinations upon return from try_call or try_call_indirect instruction. There are two possibilities: โnormal returnโ (no exception thrown), or an exceptional return corresponding to one of the listed exception tags.

The given tags are passed through to the metadata provided alongside the provided function body, and Cranelift itself does not implement an unwinder; thus, the meaning of the tags is ultimately up to the embedder of Cranelift. The tags are wrapped in Option to allow encoding a โcatch-allโ handler.

The BlockCalls must have signatures that match the targeted blocks, as usual. These calls are allowed to use BlockArg::TryCallRet in the normal-return case, with types corresponding to the signatureโs return values, and BlockArg::TryCallExn in the exceptional-return cases, with types corresponding to native machine words and an arity corresponding to the number of payload values that the calling convention and platform support. (See [isa::CallConv] for more details.)
